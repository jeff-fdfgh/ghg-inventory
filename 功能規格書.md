# 功能規格書：GHG Manager v1.6 溫室氣體盤查系統

**版本**: v1.6.1  
**日期**: 2026-04-07  
**專案路徑**: `14064-1碳盤查程式_v1.6`  
**法規依據**: ISO 14064-1、環境部 113 年版盤查登錄作業指引  
**ODS 範例對照**: `溫室氣體排放量清冊表單(範例).ods` (7.0 版)  

---

## 一、系統概述

### 1.1 目的
為事業單位提供符合環境部官方清冊格式的溫室氣體盤查自動化工具，涵蓋 ISO 14064-1 六大範疇鑑別、活動數據填報、定量盤查計算、數據品質管理，並能匯出審核用 Excel 清冊。

### 1.2 技術架構

| 項目 | 技術 |
|---|---|
| 前端 | React 19 + TypeScript + Vite 8 |
| 樣式 | Vanilla CSS (Inter 字體, glassmorphism) |
| 後端 | Node.js + Express 5 |
| 檔案上傳 | Multer (Express 中間件) |
| Excel 匯出 | SheetJS (xlsx) |
| 資料持久化 | localStorage (前端) + 本地檔案系統 (佐證) |
| 圖示庫 | lucide-react |

### 1.3 啟動方式
1. 雙擊 `啟動盤查系統.bat`
2. 自動啟動 Express server (port 3001) + 開啟瀏覽器
3. 前端 SPA 由 `dist/` 提供靜態服務

#### 2.3.3 隱藏欄位：氣體組合自動判定系統

ODS 表三包含 18 個隱藏欄位（W~AN 列）及 56 種氣體組合查詢表（AO~AR 列），用於自動判定排放源涉及的氣體名稱組合，供後續表五/表六引用。

| 隱藏欄群 | 邏輯 | 用途 |
|---|---|---|
| W~AC | 依氣體勾選顯示氣體標籤 | 顯示各氣體標籤 (CO2/CH4/...) |
| AD~AJ | 依氣體勾選產生代碼 a~g | 氣體代碼映射 (a=CO2, b=CH4, ..., g=NF3) |
| AK | 串接 AD~AJ 代碼 | 氣體組合碼 (如 "abc"=CO2+CH4+N2O) |
| AL~AN | 查表取氣體名稱 (最多3種) | 供表五/表六引用 |
| AO~AR (Row1~56) | 56 種氣體組合查詢表 | 所有可能的 1~7 氣體排列組合 |

**系統實作**: 前端依 `gases` Record 自動計算氣體組合碼並產生對應氣體名稱陣列，Excel 匯出時寫入隱藏欄。

#### 2.4.4 佐證資料管理規則

> [!IMPORTANT]
> **核心原則：每個排放源項目獨立建立專屬資料夾，嚴禁使用「未分類」。**

| # | 規則 | 說明 |
|---|---|---|
| R-01 | **禁止「未分類」資料夾** | 前端上傳時**必須**傳遞 `facility` 參數（值=設備名稱），後端不得有 `未分類` 的 fallback 邏輯 |
| R-02 | **資料夾命名** | 使用設備名稱作為資料夾名，`/` 和 `\` 替換為 `_` |
| R-03 | **檔案原始名稱** | 上傳檔案保持**原始中文檔名**，不可轉碼或重新命名 |
| R-04 | **前端顯示檔名** | 上傳成功後，在該排放源卡片中立即顯示檔案名稱列表 |
| R-05 | **「開啟資料夾」按鈕** | 只要該排放源有任何佐證檔案，必須顯示「📂 開啟資料夾」按鈕 |
| R-06 | **按鈕行為** | 點擊「開啟資料夾」→ 呼叫 `GET /open-folder/:facility` → Windows Explorer 開啟對應目錄 |
| R-07 | **持久化顯示** | 重新開啟系統後，佐證檔案清單必須仍然可見（透過 R-08 啟動掃描機制） |
| R-08 | **啟動掃描** | 系統啟動時，自動呼叫 API 掃描各設備資料夾，同步更新 `evidenceFiles` |
| R-09 | **實體存放路徑** | `{專案根目錄}/uploads/{設備名稱}/` |
| R-10 | **前端錯誤處理** | 若未提供設備名稱，前端應彈出警示並阻止上傳 |

#### 2.4.5 前端佐證區域渲染規則

> [!CAUTION]
> 佐證檔案清單和「開啟資料夾」按鈕的**資料來源一律使用 `source.evidenceFiles`**（存在 Context/localStorage 中），**不依賴元件本地 state**。這確保切換側邊欄頁籤時資料不會消失。

**渲染邏輯** (位於每個排放源展開區域底部)：

```
1. 永遠顯示「上傳佐證」按鈕
2. 若 source.evidenceFiles.length > 0：
   a. 顯示「📂 開啟資料夾」按鈕（在上傳按鈕右側）
   b. 顯示檔案名稱列表（每個檔名前顯示 📄 圖示）
   c. 點擊檔名 → 開啟該設備的資料夾
3. 上傳成功後：
   a. 呼叫 listFiles(facilityName) 取得最新檔案清單
   b. 更新 sources 中該排放源的 evidenceFiles（寫入 Context）
   c. 寫入 Context 後會自動觸發 localStorage 保存
```

**資料流**：
```
上傳 → uploadFile(facility, file) → listFiles(facility)
     → setSources(...evidenceFiles: newFiles)  ← 寫入 Context
     → localStorage 自動保存 ← useEffect 監聽 sources
     → 切換頁籤回來時 → sources 從 localStorage 載入 → evidenceFiles 仍在
```

#### 2.7.3 計算公式

- **單一排放源不確定性** (正負分別計算):
  - `U_pos = √(U_AD_pos² + U_EF_pos²)`
  - `U_neg = √(U_AD_neg² + U_EF_neg²)`
- **加權排放係數不確定性** (多氣體源，正負分別):
  - `U_EF_pos = √(Σ(E_gas × U_ef_gas_pos)²) / Σ E_gas`
  - `U_EF_neg = √(Σ(E_gas × U_ef_gas_neg)²) / Σ E_gas`
- **全廠整體不確定性** (正負分別): `U_total = √(Σ(E_i × U_i)²) / ΣE_i`
- **95% 信賴區間**: 下限 = `E × (1 - U_neg/100)`, 上限 = `E × (1 + U_pos/100)`

## 四、計算引擎

### 4.1 排放量公式

#### 4.1.1 燃料燃燒 (排放係數法，對應 PDF §2-6)

```
溫室氣體年排放量 = 年活動數據 × 低位熱值 × 單位轉換因子 × 排放係數 × GWP
```

其中：
- **年活動數據 (AD)** = totalAmount
- **低位熱值 (LHV)** = fuelLHV (kcal/unit)
- **單位轉換因子** = `4.1868 × 10⁻⁹` (kcal → TJ)
- **排放係數 (EF)** = factors[gas] (公斤/兆焦耳)
- **GWP** = 依 AR5/AR6 選擇
- 結果單位：公噸 CO2e/年

#### 4.1.2 簡化公式 (非燃料類 / 已換算係數)

```
排放量 (tCO2e) = Σ( AD × EF_gas × GWP_gas ) / 1000
```

- **EF_gas** = 已換算為單位排放係數 (公噸/unit)
- 除以 1000 換算 kg → tonne

#### 4.1.3 高低位熱值轉換 (PDF 規定)

如供應商提供高位熱值 (HHV)，需先轉換：

| 燃料型態 | 轉換比例 |
|---|---|
| 煤類（固態）及油類（液態） | LHV = **95%** × HHV |
| 氣態燃料 | LHV = **90%** × HHV |

### 4.2 GWP 版本選擇 (雙模式)

> [!IMPORTANT]
> 使用者可在「係數參考」頁面或系統設定中切換 GWP 版本。切換後所有排放源的預設 GWP 即時更新，但已自訂 GWP 的排放源（如冷媒）不受影響。

**存放於** `localStorage['ghg_gwp_version']`，值為 `'AR5'` 或 `'AR6'`。預設為 `'AR5'` (官方指引要求).

| 氣體 | AR5 (IPCC 第五次報告) | AR6 (IPCC 第六次報告) |
|---|---|---|
| CO2 | 1 | 1 |
| CH4 (非石化) | 28 | 27.0 |
| CH4 (石化) | 30 | 29.8 |
| CH4 (系統預設) | 28 | 27.9 |
| N2O | 265 | 273 |
| SF6 | 23500 | 25200 |
| NF3 | 16100 | 17400 |

若排放源有自訂 `gwpValues`，則優先使用自訂值（例如冷媒 R410A/R134a/R32 各有不同 GWP）。

## 五、後端 API (`server.js`)

| 端點 | 方法 | 說明 |
|---|---|---|
| `/upload` | POST | 上傳佐證檔案 (multipart/form-data), body **必須**含 `facility` 欄位 |
| `/files/:facility` | GET | 列出指定設備資料夾內的所有檔案名稱 |
| `/files-all` | GET | **掃描所有設備資料夾**，回傳 `{ [facility]: string[] }` 結構 |
| `/open-folder/:facility` | GET | 呼叫 Windows Explorer 開啟對應資料夾 |
| `/*` (fallback) | GET | SPA 回退，回傳 `dist/index.html` |

### 5.1 檔案命名規則
- 設備名稱中的 `/` 和 `\` 替換為 `_`
- 上傳檔案保持**原始檔名** (latin1 → utf-8 解碼)
- 實體路徑: `{專案根目錄}/uploads/{sanitized_facility_name}/{原始檔名}`

### 5.2 上傳安全規則

> [!CAUTION]
> 1. **`facility` 欄位為必填**，後端不得使用 `'未分類'` 作為 fallback。
> 2. **FormData 欄位順序**：前端必須先 `append('facility', ...)` 再 `append('file', ...)`，否則 multer 在 `destination` callback 時讀不到 `facility`。

```javascript
// 前端 (fileManager.ts) — 正確順序：
const fd = new FormData();
fd.append('facility', safeName);  // ← 必須在 file 之前
fd.append('file', file);

// 後端 (server.js) — 包裝 multer error handler：
app.post('/upload', (req, res) => {
    upload.single('file')(req, res, (err) => {
        if (err) return res.status(500).json({ error: err.message });
        if (!req.file) return res.status(400).json({ error: '未收到檔案' });
        if (!req.body.facility) return res.status(400).json({ error: '必須指定設備名稱' });
        res.json({ success: true, filename: req.file.filename, facility: req.body.facility });
    });
});
```

### 5.3 啟動掃描 API (`/files-all`)

系統啟動時前端呼叫此 API，一次取得所有設備的佐證檔案清單：

```javascript
app.get('/files-all', (req, res) => {
    if (!fs.existsSync(UPLOAD_ROOT)) return res.json({});
    const result = {};
    fs.readdirSync(UPLOAD_ROOT).forEach(folder => {
        const dir = path.join(UPLOAD_ROOT, folder);
        if (fs.statSync(dir).isDirectory()) {
            result[folder] = fs.readdirSync(dir).filter(f => !f.startsWith('.'));
        }
    });
    res.json(result);
});
```
